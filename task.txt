CHANGES MADE:
1. SaleDao.java (getMaxInvoiceNumber method):
   - Added explicit transaction handling with session.beginTransaction()
   - Changed session resource management to use try-finally pattern
   - Increased query limit from 200 to 500 invoices
   - Added detailed logging to track invoice number discovery
   - Added proper transaction commit/rollback
   - Ensures the query reads committed data from all previous sales

ROOT CAUSE:
The issue was that getMaxInvoiceNumber() was opening a session without explicit 
transaction handling, which could result in reading stale or inconsistent data. 
This caused the method to always return 0, leading to repeated "INV-001" generation.

THE FIX:
- Added explicit transaction boundaries to ensure data consistency
- The query now properly sees all committed invoices from the database
- Next invoice will be max(existing invoices) + 1
- All existing INV-001 invoices are preserved (as requested)
- From now on, invoices will properly increment: INV-002, INV-003, etc.

HOW TO TEST:
1. Stop the POS application if running
2. Start the POS with: start-pos.bat
3. Create a new sale
4. Check the console output for: "[SaleDao] Returning max invoice number: X"
5. The new invoice should be INV-00X where X is one more than the highest
6. Create another sale to verify it increments correctly

WHAT TO EXPECT:
- Console will show: "[SaleDao] Found N invoice numbers starting with INV-"
- Console will show: "[SaleDao] New max found: INV-XXX -> XXX"
- Console will show: "[SaleDao] Returning max invoice number: XXX"
- Console will show: "[SalesService] Generated invoice number: INV-XXX"
- Each new sale should have a unique, incrementing invoice number
